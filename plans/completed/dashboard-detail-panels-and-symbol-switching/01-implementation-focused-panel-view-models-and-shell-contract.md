# Implementation Step 1: Focused Panel View Models And Shell Contract

## Requirements And Scope

- Replace the generic focused-section placeholder contract with a shell-facing panel model that is specific enough to render overview, microstructure, derivatives, and health/regime content.
- Keep all market interpretation service-owned; the mapper may format timestamps, choose compact display labels, and assemble presentational groupings only.
- Reuse the existing `useDashboardData` and `dashboardStateMapper` seam so symbol switching continues to run through one bounded data flow.

## Target Repo Areas

- `apps/web/src/features/dashboard-shell/model`
- `apps/web/src/features/dashboard-state`
- `apps/web/src/pages/dashboard`

## Implementation Notes

- Introduce focused-panel view-model types in the dashboard shell model, for example:
  - `DashboardFocusedPanel`
  - panel metric rows or label/value pairs
  - optional callout or refresh-note fields
- Keep the top-level shell contract explicit: summary strip and status rail stay unchanged, while the detail region receives focused panel data instead of generic placeholders.
- Extend `deriveDashboardViewModel` so it produces the focused-panel object from existing `DashboardSymbolStateContract` and `DashboardGlobalStateContract` records.
- Preserve the bounded trust vocabulary already adopted by the dashboard: `loading`, `ready`, `stale`, `degraded`, `unavailable`.
- Use the existing record error and last-success metadata to generate panel-local fallback notes when cached last-known-good data is shown after refresh failure.
- Keep the derivatives panel generated by the mapper even if the underlying contract remains unavailable; this lets the shell treat all four panel slots uniformly.

## Data Mapping Rules

- Overview panel should derive from the focused symbol current-state payload only.
- Microstructure panel should derive from the focused symbol bucket and recent-context payload only.
- Health/regime panel may combine focused symbol and global current-state payloads, but only by displaying existing service-owned fields.
- Derivatives panel should not invent fields; it should produce an explicit unavailable panel contract when no derivatives surface exists.

## Unit Test Expectations

- Extend `apps/web/src/features/dashboard-state/dashboardStateMapper.test.ts` to cover:
  - focused BTC and ETH panel view-model generation
  - cached last-known-good panel notes after refresh failure
  - explicit derivatives unavailable panel output
  - health/regime panel composition from symbol plus global payloads
  - invalid or missing focused data staying in `loading` or `unavailable` rather than producing guessed content

## Summary

This step creates the implementation seam the rest of the feature depends on: one focused-panel contract that stays presentational, preserves the existing trust model, and keeps panel rendering independent from raw payload shape.
